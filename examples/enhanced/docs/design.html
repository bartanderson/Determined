Commonplace — Architecture & Determined Surface Map

Living design document. Update as the app and the tool evolve. Located at Determined/examples/commonplace/docs/design.html

App Modules

app.pyFlask factory. Config wiring. Blueprint registration.
config.pyDB path, LLM endpoint, feature flags (tagging on/off)
models/
entry.pyEntry: type (url/snippet/note/code), content, metadata, tags, created_at
tag.pyTag: name, source (manual | llm). Many-to-many with Entry.
connection.pyDirected link between two entries. Type: related/contradicts/extends.
routes/
capture.pyPOST /capture. Calls extractor, tagger, stores entry. call graph
browse.pyGET / and /entry/<id>. Renders entry + connections.
search.pyGET /search. Delegates to searcher service. reason
api.pyJSON API -- future JS frontend hook.
services/
extractor.pyFetch URL, parse title/meta, extract content. Doing too much. frontier reason
tagger.pysuggest_tags(content) returns []. LLM stub. Call-on-write vs lazy is open. frontier
linker.pyfind_connections(entry_id) returns []. Inference stub. frontier
searcher.pyText search now, semantic later. Calls both models/ and storage/. reason
storage/
db.pySQLite connect, schema init, migrations.
queries.pyQuery helpers. Also called by searcher -- mixed abstraction level.
utils/
url.pyNormalize, validate URL. Also called from capture route -- duplication.
text.pyTruncation, cleaning, excerpt generation.

Planted Design Tensions -- Reason targets

  • Extractor does too muchfetch + parse + metadata extraction in one module. Split or not?
  • URL validation duplicationurl.py (utils) AND capture.py both validate. Who owns it?
  • Tagger: eager vs lazycall on write (slow capture) or on demand (stale tags)? Open.
  • Connection directionalityconnection.py model does not enforce bidirectional consistency.
  • Searcher abstraction leakcalls models/ and storage/ -- bypasses its own layer boundary.

Determined Surfaces Exercised

Frontier3 real stubs (tagger, linker, extractor full-content). Frontier tab immediately useful.
Call Graphcapture route to extractor to db to entry. Real multi-hop chain to trace.
Reason5 planted tensions, each a real Reason question. Not contrived.
Projectextractor.py and tagger.py are obvious Project candidates.

UI Issues Commonplace Will Force

Panel size -- Navigate opens file into tiny right panel
12+ files across subdirs. You will hit this on day one of loading the corpus. Forces the holistic UI redesign to have a concrete driver.
Context switching -- 6 tab hops for one analysis
Frontier to Editor to call graph to Reason to back. Editor needs to be the hub with contextual overlays, not a secondary panel.
Navigation breadcrumbs
Clicking a caller in the call tree should navigate the editor, not open a second panel. Where-did-I-come-from gets lost fast in a 15-file project.

The Ideal Analysis Flow (one continuous thread, no tab gymnastics)

1Load corpusFrontier tab
->
2Click tagger stubEditor opens
->
3"Who calls this?"Call graph inline
->
4Click ReasonPanel alongside editor
->
5Click ProjectProjection overlay
->
6Click callerEditor navigates, breadcrumb back
Current UI breaks at step 3 (tiny panel) and step 6 (no breadcrumb). Commonplace makes this viscerally obvious on first use.
